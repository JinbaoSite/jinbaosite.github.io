# 字节对编码(BPE):一种简单而高效的开放词表分词方法

如果你用过 GPT、LLaMA 或者任何一个现代大语言模型,你输入的每一句话在进入模型之前,都会先被切成一个个"token"。而决定怎么切的,往往就是一个出人意料地简单、却异常有效的算法——**BPE(Byte Pair Encoding,字节对编码)**。

## 1. 为什么分词这么难

模型不认识文字,只认识数字。所以第一步永远是把文本映射成一串整数 ID。问题在于:**以什么为单位来切?**

最直觉的两种做法各有硬伤。

**按词切(word-level)。** 把 "I love cats" 切成 `["I", "love", "cats"]`,简单直观。但词表会爆炸——英语常用词就有几十万,加上各种时态、复数、拼写变体更是无穷无尽。更糟的是**未登录词(OOV)问题**:只要训练时没见过的词,比如一个新造的网络热词或拼写错误,模型就只能用一个统一的 `<UNK>` 来表示,信息全丢了。

**按字符切(char-level)。** 把每个字母当一个 token,词表小到只有几十个,也彻底没有 OOV 问题。但代价是序列变得极长,而且单个字符几乎不携带语义,模型得费很大劲才能从字符重新"拼"出词义。

我们真正想要的,是介于两者之间的东西:常见的完整词保留为一个 token,生僻或没见过的词则拆成有意义的**子词(subword)**片段。比如把 "tokenization" 拆成 `token` + `ization`,把 "unhappiness" 拆成 `un` + `happiness`。这样既控制了词表大小,又能优雅地处理没见过的组合。

BPE 正是用一种极其朴素的方式,自动学出这样一套子词切分。

## 2. BPE 的前世今生

有意思的是,BPE 最初根本不是为 NLP 设计的。它在 1994 年由 Philip Gage 提出,是一个**数据压缩算法**:思路是反复找出文件里出现最频繁的一对相邻字节,用一个没用过的新字节去替换它,如此循环,从而压缩数据。

2016 年,Sennrich 等人在机器翻译的论文里把这个老算法"借"了过来,只是把"字节"换成了"字符",把"压缩"的目标换成了"构建子词词表"。从此 BPE 成了神经网络文本处理的标配,GPT 系列、RoBERTa、LLaMA 等模型用的都是它或它的变体。

核心思想一句话就能说清:

> **从最小的单位开始,反复把"最常一起出现的相邻片段"合并成一个新片段,直到词表达到目标大小。**


## 3. 训练阶段

假设我们统计出语料里有这几个词及其出现次数,并且在每个词末尾加一个特殊标记 `_` 表示"词的边界"(这样模型才能区分 `est` 在词中间还是词尾):

```
5   l o w _
2   l o w e s t _
6   n e w e r _
3   w i d e r _
2   n e w _
```

每个词一开始都被打散成单个字符。现在开始迭代。

**第 1 轮:统计所有相邻字符对的频率。**

我们扫描每个词的相邻对,按词频加权累加。挑几个关键的:

- `(e, r)`:出现在 `newer`(6 次)和 `wider`(3 次)→ 共 **9**
- `(r, _)`:同样出现在 `newer` 和 `wider` → 共 **9**
- `(n, e)`:出现在 `newer`(6)和 `new`(2)→ 共 **8**
- `(l, o)`:出现在 `low`(5)和 `lowest`(2)→ 共 **7**

最高频的是 `(e, r)`,出现 9 次(`(r,_)` 同为 9,平局时按实现约定取其一)。于是**第一次合并:`e` + `r` → `er`**。

语料更新为:

```
5   l o w _
2   l o w e s t _
6   n e w er _
3   w i d er _
2   n e w _
```

**第 2 轮:重新统计。** 现在 `(er, _)` 出现在 `newer`(6)和 `wider`(3)→ 共 9,成为最高频。**合并 `er` + `_` → `er_`**:

```
5   l o w _
2   l o w e s t _
6   n e w er_
3   w i d er_
2   n e w _
```

**继续迭代。** 后续几轮会依次得到这些合并规则(你可以自己验证):

| 轮次 | 合并的对 | 新片段 |
|------|----------|--------|
| 3 | `n` + `e` | `ne` |
| 4 | `ne` + `w` | `new` |
| 5 | `l` + `o` | `lo` |
| 6 | `lo` + `w` | `low` |
| 7 | `new` + `er_` | `newer_` |
| 8 | `low` + `_` | `low_` |

注意一个关键点:**合并出来的片段会成为下一轮的"原子",可以参与更大的合并**。所以 `n→ne→new→newer_` 是层层累积出来的。

训练结束后,我们真正得到的产物是两样东西:

1. **一张词表(vocabulary)**:所有出现过的片段,从单字符到 `low_`、`newer_` 这样的整词。
2. **一份有序的合并规则表(merge rules)**:就是上面那张表,**顺序至关重要**,编码新文本时要严格按这个顺序应用。

合并的轮数(`num_merges`)是唯一的超参数,它直接决定最终词表的大小——轮数越多,词表越大、片段越长越接近整词。


## 4. 应用阶段:给新文本分词

训练好的合并规则怎么用在新句子上?流程是:把新词同样打散成字符,然后**按训练时学到的顺序**,依次尝试应用每一条合并规则。

比如来了个训练时没完整见过的词 `newest`:

```
起始:     n e w e s t _
应用 e+r:  (没有 er,跳过)
...
应用 n+e:  ne w e s t _
应用 ne+w: new e s t _
...
```

最终它会被切成 `new` + `e` + `s` + `t` + `_` 之类的片段。**关键在于:即使 `newest` 从没在训练集里出现过,我们依然能用学过的 `new` 把它的大部分语义保留下来**,而不是粗暴地丢给一个 `<UNK>`。这就是 BPE 解决 OOV 问题的方式。


## 5. Python 实现

下面是 Sennrich 论文里那段经典实现的精简版,完全可以直接跑。

```python
import re
import collections


def get_stats(vocab):
    """统计所有相邻片段对的加权频率"""
    pairs = collections.defaultdict(int)
    for word, freq in vocab.items():
        symbols = word.split()
        for i in range(len(symbols) - 1):
            pairs[symbols[i], symbols[i + 1]] += freq
    return pairs


def merge_vocab(pair, vocab_in):
    """把指定的对在整个语料里合并成一个片段"""
    vocab_out = {}
    bigram = re.escape(' '.join(pair))
    # 用负向断言确保只匹配完整片段,不会切到别的词中间
    pattern = re.compile(r'(?<!\S)' + bigram + r'(?!\S)')
    for word in vocab_in:
        merged = pattern.sub(''.join(pair), word)
        vocab_out[merged] = vocab_in[word]
    return vocab_out


# 初始语料:每个词打散成字符,末尾加 </w> 边界标记
vocab = {
    'l o w </w>': 5,
    'l o w e s t </w>': 2,
    'n e w e r </w>': 6,
    'w i d e r </w>': 3,
    'n e w </w>': 2,
}

num_merges = 8
merge_rules = []

for i in range(num_merges):
    pairs = get_stats(vocab)
    if not pairs:
        break
    best = max(pairs, key=pairs.get)   # 取最高频的对
    vocab = merge_vocab(best, vocab)
    merge_rules.append(best)
    print(f"第 {i + 1} 轮合并: {best}  ->  {''.join(best)}")

print("\n学到的合并规则(有序):")
print(merge_rules)
```

运行后会打印出每一轮选择合并的对,和我们手算的结果一致。代码只有三个要点:

- `get_stats` 负责"数对子",这是每轮决策的依据。
- `merge_vocab` 用正则把选中的对在全语料里替换掉,`(?<!\S)` 和 `(?!\S)` 两个断言保证只匹配被空格隔开的完整片段。
- 主循环就是不断"统计 → 选最高频 → 合并",重复 `num_merges` 次。

短短几十行,就是现代分词器的核心骨架。

## 6. Byte-Level BPE:真实大模型用的版本

上面讲的是字符级 BPE,但 GPT-2 之后的主流模型用的是一个更巧妙的变体:**字节级 BPE(Byte-Level BPE)**。

区别在于起点:它不从 Unicode 字符开始,而是从**原始字节**开始。任何文本都能编码成 UTF-8 字节序列,而字节只有 256 种可能值。这带来一个杀手级好处:

> **基础词表只需 256 个字节,就能表示世界上任何语言、任何 emoji、任何符号——彻底、永久地消灭了 OOV 问题。**

无论是中文、阿拉伯文,还是 `🦝`,在字节层面都只是若干个 0–255 的数字,BPE 照样能在上面学合并规则。这就是为什么现代模型不管你输入什么稀奇古怪的字符都不会"看不懂"。代价是某些非拉丁文字(比如中文)会被切成更多 token,因为一个汉字通常占 3 个 UTF-8 字节。

## 7. 优缺点

BPE 能成为事实标准,靠的是一组漂亮的权衡。

**它的优势在于:** 词表大小完全可控(由合并轮数决定);通过子词共享大幅提升了对罕见词和词形变化的处理能力;字节级版本从根本上杜绝了 OOV;算法本身简单、训练快、纯靠数据驱动,不需要任何语言学规则或人工标注。

**它也有局限:** 合并完全由频率驱动,切出来的片段不保证有语言学意义(可能把一个词切在很别扭的位置);训练和编码都是贪心的,不保证全局最优;对形态丰富的语言,切分效果有时不如基于语言学的方法。

正因如此,后来也出现了一些"近亲":**WordPiece**(BERT 用,合并时不看频率而看似然增益)、**Unigram LM**(从一个大词表反向裁剪),以及把它们打包好的工业级库 **SentencePiece**。但它们的灵魂,和 1994 年那个压缩算法是一脉相承的。

## 8. 结语

BPE 的迷人之处,就在于它用一个朴素到近乎"笨"的规则——**反复合并最常见的相邻对**——优雅地解决了分词里最棘手的取舍。它不懂语法,不懂语义,只会数数和合并,却恰好踩在了词表大小、序列长度和泛化能力之间的甜蜜点上。

下次当你看到模型把一个长单词切成几块时,你就知道,那背后是一个跑了几万轮"合并最高频对"的简单循环,在默默工作。
